# blurkit docs

These docs describe the package as it exists today: explicit runtime entrypoints, a browser-safe encode path, the current CLI surface, manifest helpers, and the result shapes returned by the library.

```ts title="scripts/encode-hero.ts"
import { encode } from 'blurkit/node'

const result = await encode('./public/hero.jpg', {
  algorithm: 'blurhash',
  size: 32,
})
```

:::note
Use runtime-specific imports in production code. `blurkit/node`, `blurkit/browser`, and `blurkit/edge` make the supported environment explicit and keep bundlers honest.
:::

## What the package covers

- Runtime-specific imports:
  - `blurkit/node`
  - `blurkit/browser`
  - `blurkit/edge`
- A root convenience import:
  - `blurkit`
- Batch encoding through `encodeMany()`
- A Node memory cache helper through `blurkit/node`
- Manifest helpers through the root package and Node entrypoint
- A CLI for single-image output and folder manifests

## What `encode()` returns

Every successful encode call resolves to a `BlurResult` containing:

- A ready-to-use `dataURL`
- The raw placeholder `hash`
- The `algorithm` used to generate it
- The placeholder `width` and `height`
- Optional source `meta` with original image dimensions and format

## Runtime input coverage

```text title="runtime-coverage.txt"
Node    : local path, remote URL, Blob, File, ArrayBuffer

Edge    : remote URL, Blob, File, ArrayBuffer

Typical fit:

Browser -> client-side uploads and private local previews
Edge    -> platform fetch pipelines without sharp
```
